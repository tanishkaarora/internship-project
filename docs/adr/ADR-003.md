# ADR-003: Choosing Streamlit Monolith vs Decoupled FastAPI and React/TypeScript Frontend

## Context

For the AI Retail Decision Copilot, we required an interactive web interface that provides:
1. File uploaders for tabular data (CSV/Excel) and PDFs.
2. Dashboard components showing key metrics (sales, averages) and charts (trends, category comparisons).
3. A conversational chat window supporting multi-turn assistant messaging with citations.

We evaluated two architectural approaches for constructing the web application:
1. **Unified Streamlit Monolith:** A single Python script driving both the backend logic (orchestrated via LangGraph) and rendering the UI layout.
2. **Decoupled API + SPA Architecture:** A FastAPI backend exposing REST/WebSocket endpoints and a React/TypeScript frontend (built with Vite and TailwindCSS) executing the user interface.

## Decision

We decided to build the MVP as a **Unified Streamlit Monolith** for Phase 1. 

All user inputs, file processors, interactive Plotly visualizations, and the LangGraph conversation agent run within a single Streamlit process. This is maintained in the root `streamlit_app.py` script.

## Consequences

### Positive

* **Rapid Frontend Prototyping:** Streamlit enables writing complete user interfaces in pure Python. By eliminating frontend boilerplate (React component structures, build tooling like Vite, state stores like Redux, and Tailwind styling classes), the entire interface is written in under 250 lines of code.
* **Simplified State Sharing:** Since the UI and backend run in the same Python process, we can store large Pandas DataFrames directly in the `st.session_state` dict. This bypasses the need to serialize and transmit multi-megabyte CSV data over HTTP networks during user queries.
* **Faster Setup time:** The user only has to run one command (`streamlit run streamlit_app.py`) to launch the application. This ensures reviewers can clone and run the app in less than 20 minutes.
* **Built-in Plotly Support:** Streamlit natively supports rendering interactive, hover-responsive Plotly figures using `st.plotly_chart`.

### Negative

* **Strict Page Refresh Model:** Streamlit's execution model re-runs the entire script from top to bottom on every user interaction (e.g., button clicks, typing in chat). This requires careful session state gating to prevent reloading files or losing chat history.
* **Limited UI Customization:** While Streamlit supports custom CSS styling, developers are constrained by the framework's native layout widgets (columns, sidebars, tabs). Building bespoke dashboard structures or custom visual components is much harder than in a pure HTML/React setup.
* **Scalability Bottlenecks:** A monolithic design binds user sessions to a single python process, making it difficult to scale the app to handle thousands of concurrent users in production.

### Trade-offs

We chose to accept the layout limitations of Streamlit to focus our development time on the core data engineering and AI orchestration layers (LangGraph, FAISS, Pandas). The monolithic approach ensures a highly presentable and fully functioning app can be built quickly, while establishing a clear roadmap to migrate to a decoupled FastAPI + React architecture during Phase 2 (3rd year expansion).

## Alternatives Considered

* **FastAPI + React/TypeScript SPA:** While this architecture is highly scalable and standard for production environments, it was rejected for this phase because it introduces significant build complexity (running separate node and python servers, establishing cross-origin API headers, and configuring multi-file upload endpoints), which would push the setup time for a reviewer well past the 20-minute threshold.
* **Gradio Monolith:** Similar to Streamlit but commonly used in machine learning spaces. We chose Streamlit over Gradio because Streamlit's layouts are better suited for data-rich dashboards and support native tabs, column splits, and complex KPI metric cards.
