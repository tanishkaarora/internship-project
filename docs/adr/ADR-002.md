# ADR-002: Choosing FAISS over ChromaDB for Lightweight In-Memory Vector Store

## Context

The AI Retail Decision Copilot supports unstructured document QA (RAG) by parsing PDFs, splitting their content into overlapping text chunks, generating semantic embeddings, and indexing them locally. 

When choosing a vector database for local execution, we evaluated two primary candidates:
1. **FAISS (Facebook AI Similarity Search):** A library for efficient similarity search and clustering of dense vectors.
2. **ChromaDB:** A dedicated, developer-focused open-source vector database.

We needed a solution that:
* Requires zero infrastructure setup (no Docker containers or external servers to launch).
* Can run entirely in-memory within the Streamlit process context to ensure seamless local developer setup.
* Has minimal Python dependency footprints to avoid installation failures across different developer platforms (Windows/macOS/Linux).
* Performs rapid sub-second similarity searches for small single-user document sizes.

## Decision

We chose **FAISS (cpu-only variant `faiss-cpu`)** as the local vector indexing library, orchestrated via the LangChain `FAISS` community integration.

FAISS serves as an in-memory vector index that does not require starting any database daemon process. When a user uploads a PDF, the text chunks are embedded and loaded into memory instantly.

## Consequences

### Positive

* **Zero Infrastructure Overhead:** FAISS is a lightweight Python package. Reviewers and developers can run the project without having to spin up Docker containers, manage background services, or configure database network ports.
* **Instantaneous Search Latency:** Since FAISS is written in optimized C++ and runs entirely in the Python process memory space, similarity searches complete in sub-millisecond times for typical user documents.
* **Simple Serialization:** FAISS indexes can be serialized to a single binary file on disk or held directly in the Streamlit session state, simplifying state management.
* **Stable Windows Installation:** The `faiss-cpu` package compiles cleanly on Windows platforms (unlike some SQLite/C++ bindings in early Chroma versions which historically caused local install errors on Windows systems).

### Negative

* **No Persistent Database Server:** FAISS is a library, not a standalone database engine. It does not support native multi-tenant connection routing, enterprise access control, or remote API execution out of the box.
* **Limited Metadata Filtering:** While FAISS supports basic metadata storage, it is not optimized for complex SQL-like metadata filtering or structured joins compared to specialized database services.

### Trade-offs

We accepted the lack of enterprise features because the current scope of this project is a single-user local prototype (Phase 1). For this target profile, minimizing setup complexity and ensuring that a reviewer can successfully clone and run the app in <20 minutes is a critical priority, making FAISS the superior choice.

## Alternatives Considered

* **ChromaDB:** Chroma is highly developer-friendly and offers advanced query filters. However, it was rejected for the Phase 1 MVP because its binary dependencies (e.g. `hnswlib` compilation requirements, specific SQLite versions) frequently cause install-time failures on Windows developer machines, violating our requirement for a smooth 20-minute setup.
* **In-Memory Numpy cosine-similarity:** Writing a raw numpy-based similarity matrix would have eliminated dependencies entirely. This was rejected because using a standard vector library like FAISS ensures a cleaner transition to 3rd-year production-grade engines and demonstrates industry-standard package orchestration.
